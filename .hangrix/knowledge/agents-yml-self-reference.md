# `.hangrix/agents.yml` lives in three places at once

This repo IS the Hangrix platform, so the agent-config schema is both the *implementation* and a *consumer*. A change to the schema MUST propagate through three artefacts in the same commit, or one of them will quietly drift and the next change will surface the drift as a bug:

1. **Parser** — `apps/hangrix/internal/agentsconfig/{host.go, parse_host.go, normalize.go, role.go, triggers.go, mentions.go}`. The Go types here are the runtime contract. Round-trip tests live at `parse_host_test.go`, `triggers_test.go`, `mentions_test.go`. After editing `.hangrix/agents.yml` (or a role file), confirm it still parses with `go test ./apps/hangrix/internal/agentsconfig/...`.
2. **JSON Schema** — `docs/agents.schema.json`. External tooling (editors, validators) reads this. Keep field names, enum values, and `oneOf` mutex blocks in lockstep with the parser.
3. **Starter template** — `apps/hangrix/internal/modules/repo/templates/initial/.hangrix/agents.yml`. This is what the seeder hands fresh host repos. It is parsed at server startup time, so a schema drift here means every new host repo lands broken on day one.

The reference doc `docs/agent-config.md` is the human-readable view of the same contract — update it too when behaviour changes, not just when fields move.

**Forward-compat footgun.** The parser silently drops unknown keys (so an older agent server can read a newer host file without bricking). A typo in a key name therefore parses cleanly and does nothing — no error, no warning. If a trigger fires more or less often than expected, the first thing to check is whether you mis-spelled the canonical key.

**`can:` / `not:` scope.** These ACL lists only filter **agent HTTP API tools** (the network-served `issue_read`, `issue_comment`, `issue_review_vote`, `issue_checks`, `issue_merge`, `issue_close`, `issue_children`, `roster_list`, `contribution_list`, `contribution_read`). The filter lives at [apps/hangrix/internal/modules/agent_api/service/can.go:74-96](apps/hangrix/internal/modules/agent_api/service/can.go#L74-L96) and is only consulted on the agent HTTP API code path. The agent binary's **built-in** catalogue — `read`, `write`, `edit`, `glob`, `grep`, `bash`, `webfetch`, `compact_session` — is baked in and always available; the agent runtime never asks `can:` about them. Putting built-in names in `can:` is dead weight (silently ignored), and putting them in `not:` does not actually block them. Read-only reviewer discipline is therefore a prompt convention, not a runtime guarantee.

**`reasoning_effort:` is intentionally not enum-validated.** The parser accepts any string so a newer upstream enum value (`max`, `extra_high`, vendor-specific labels) can be filled without an adapter change. Only the canonical four — `minimal` / `low` / `medium` / `high` — drive the Anthropic adapter's `thinking.budget_tokens` translation at [apps/hangrix/internal/modules/llm_proxy/upstream/anthropic.go:202-217](apps/hangrix/internal/modules/llm_proxy/upstream/anthropic.go#L202-L217); other values pass through verbatim to OpenAI-compat and disable thinking on Anthropic (the adapter refuses to guess a budget that could exceed `max_tokens`). If you actually want Anthropic thinking, use one of the four canonical values.

This very file (`.hangrix/agents.yml` at the repo root) is the local concrete instance — the team for working *on* the platform itself, exercised by every agent woken by an issue here.

**`container.secrets` has been removed (issue #31).** The `secrets:` array (name-only list of repo secrets) is gone from the schema, parser, starter template, and docs. It is replaced by repo-scoped **variables** and **secrets** managed through the repo settings page (`/api/repos/{owner}/{name}/variables`). A plain `variable` stores its value in the clear; a `secret` encrypts the value server-side and never reveals it after creation. Both are referenced from `container.env` values via `${VAR_NAME}` whole-value expansion, performed by the runner at session start — not at parse time. The `agentsconfig` parser accepts `${...}` syntax in env values as opaque strings; variable resolution and expansion are the runner's responsibility.
